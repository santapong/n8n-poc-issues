1 · The one-paragraph answer

When a workflow runs, data piles onto the V8 heap. When it finishes, that memory becomes dead — but V8 does not free it on a timer or when the run ends. V8 frees dead memory only when it feels memory pressure (a big enough new allocation) or on an occasional idle sweep. So an idle n8n holds its heap at the high-water mark — it does not fall back to baseline by itself. It falls when the next big run forces a major GC, when you force one, or on restart. This is normal V8 behaviour, not a leak.

2 · Three numbers, not one

"Heap memory" is really three nested figures. Confusing them is the root of most memory questions.

3 · The sawtooth & the high-water mark

Over a day, heap_used does not return to baseline between runs. It traces a coarse sawtooth that parks near the high-water mark — the biggest thing the process has handled — and only drops when pressure forces a collection.

4 · Why it STAYS — GC is pressure-driven, not time-driven

V8 runs two kinds of collection: cheap scavenges (young objects) constantly, and expensive mark-compact (the old space, where a base64 string or a big array lives) only under old-space pressure. A finished run leaves its data dead in old space — but if nothing new is allocated, no mark-compact runs, so the dead data lingers.

Evidence — idle does not collect; tiny activity does not either

baseline                         heap_used = 144 MB
upload max.pdf (120 MB)          heap_used = 461 MB   # base64 on heap
… idle 40 s, no activity …       heap_used = 465 MB   # STAYS — not collected
10 tiny uploads (598 B each)     heap_used = 468 MB   # too little pressure → still no major GC
Key mechanism A finished execution does not trigger a major GC. Neither does idle time, nor small allocations. The dead data is collectible — V8 just hasn't been given a reason to collect it yet.

5 · When & how it goes DOWN

Automatically — a big-enough next allocation forces the major GC

stuck idle                       heap_used = 470 MB
upload medium.pdf (25 MB)  →     heap_used = 212 MB   # t+4s: major GC freed the old 160 MB blob

The drop is triggered by the new allocation's pressure, not by a clock. So in a busy system the next big run cleans up the previous one; in an idle system the dead data can sit for a long time.

On demand — force a full GC (built into the monitor)

stuck idle                       heap_used = 461 MB
Force GC (SIGUSR2 snapshot)  →   heap_used = 141 MB   # full mark-compact, back to floor

A V8 heap snapshot performs a full mark-compact as a side effect. The ⟲ Force GC button (and /api/gc) triggers it and deletes the snapshot — freeing all dead data on demand.

6 · High-water mark vs a real leak

  • Not a leak: RSS parks near the peak all day, but heap_used returns to ~150 MB whenever a GC runs. Reusable committed memory.
  • Leak: the value right after a forced GC keeps rising across restarts of activity — something is holding references it shouldn't.

7 · UI, manual runs & common myths

BeliefReality
"Closing the web UI frees the heap"No. The UI is your browser; the heap is the server. Closing a tab frees browser memory, not server memory.
"Navigating to another page frees it"No. Same reason. (Exception: opening a big execution in the editor makes the server load that data into its heap to send it — leaving the page stops re-fetching, then GC frees it.)
"Memory only moves on manual runs"No. Any execution moves it. A manual run executes on main (and animates the canvas); a production webhook executes on the worker (shows in the Executions tab, no canvas).
"It should drop right after the run finishes"No. Finishing a run doesn't trigger a major GC (§4).

8 · Running 24/7 — recommendations

  1. Keep the high-water mark low (best): move large binary off-heap (filesystem/s3) and cap upload size + concurrency, so the heap never climbs to 465 MB in the first place.
  2. Recycle the process to release RSS: a scheduled restart (example recycler service in docker-compose.yml) or restart-after-N-executions. You already have restart: unless-stopped.
  3. Size the heap from the sawtooth peak: --max-old-space-size ≥ concurrency × max_MB × 2.3, container memory above that.
  4. Monitor the floor, not the peak: alert on heap_used after GC trending up over days (leak), and on container RSS approaching the limit (imminent OOM).
  5. Force GC only as a stopgap: the /api/gc button is great for demos and manual relief, not a substitute for (1)–(3).

9 · Tools & how to reproduce

  • heap-monitor (:8888) — live heap_used + heap_total + RSS + per-space bar; ⟲ Force GC and ◍ Capture raw heap (with timed capture & PDF filter).
  • /api/gc?node=main — force a full GC, returns before → after (freed).
  • recycler service (commented in compose) — periodic restart for 24/7.
# watch it stay, then force it down
curl -F "file=@max.pdf" http://localhost:8080/webhook/upload        # heap → ~465 MB
# …wait, watch http://localhost:8888 — it stays…
curl -s "http://localhost:8888/api/gc?node=main"                    # → freed ~160-320 MB

# instant full reset (heap_used AND RSS)
docker compose restart n8n-main
