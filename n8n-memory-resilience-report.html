Memory Resilience Strategies — Evidence & Evaluation

Three proposed strategies for the web node, each tested against live measurements. n8n 2.28.6.
multi-instance failoverforce-free memory auto-restartmeasured, not assumed

Verdict at a glance

Proposed strategyVerdictWhat the evidence shows
① Multi-instance, switch on max, wait for memory to reduceHALF-RIGHTMulti-instance is good; "wait to reduce" fails — idle memory plateaus above baseline and is non-deterministic (Evidence A). And spikes are too fast to switch before (Evidence D).
② Remove directly from memory (force-free)WORKS LIMITEDForce GC deterministically drops heap to the floor (Evidence B) — but only dead data, not the live request causing the OOM.
③ Auto-restartNEEDED NOT UX-SAFE ALONERecovers automatically, but costs a ~6.5 s outage window with failed requests (Evidence C).
Bottom line Resilience (multi-instance + auto-restart) is worth doing, but it does not self-heal by "waiting" — idle memory doesn't reliably shrink, so you must recycle (force GC or restart). And all of it is a safety net, not a substitute for preventing the OOM.

Evidence A — does a drained/idle instance reduce on its own?

Uploaded a 120 MB PDF to spike the heap, then left the node completely idle and sampled both heap_used and container RSS:

Stepheap_usedRSS
baseline152 MB318 MB
after 120 MB upload462 MB711 MB
idle +15 s302 MB392 MB
idle +30 s302 MB393 MB
idle +45 s303 MB393 MB
Kills "switch and wait" You cannot rely on a drained instance to free its memory — it plateaus above baseline and the timing is unpredictable. The correct move is drain → recycle (force GC / restart), not drain → wait.

Evidence B — can you remove memory directly (force-free)?

From that stuck plateau, triggered a full GC via the monitor's ⟲ Force GC (/api/gc):

Stepheap_usedRSS
stuck plateau (before GC)302 MB393 MB
after Force GC142 MB289 MB

Yes — it works, deterministically: heap_used dropped to the floor (302→142) and RSS came down too (393→289), on demand, in seconds. But two hard limits:

  • It only frees dead objects. The base64 string of an in-flight upload is live — you cannot force-free the very data that is causing the OOM. So this does not rescue you during the bad request.
  • It is a stop-the-world pause (mark-compact) — running it under load adds latency.
Verdict A great manual / scheduled relief valve (built into the lab), not a load-time rescue.

Evidence C — is auto-restart good for UX?

Restarted n8n-main and probed /metrics every 0.5 s to measure the unavailability window:

# requests during the restart
non-200 responses  : 13 consecutive
downtime window    : ~6.5 s   # web node unavailable; in-flight requests fail
then               : 200 OK — service back
Verdict Necessary safety net, but not UX-safe on its own. It only becomes graceful behind multiple instances — then only the crashing node's requests fail while the others serve. A restart does cleanly reset heap toward the floor — the one thing idle waiting never achieves.

Evidence D — can you detect & switch before the crash?

Fine-grained sampling during an upload (every ~150 ms):

t=0.6s  heap = 144 MB
t=0.9s  heap = 280 MB
t=1.1s  heap = 464 MB     # +320 MB in ~0.5 s
Too fast to react The heap crosses from safe to near-cap in ~0.5 s. A load-balancer health check polling every few seconds cannot drain an instance before the spike crashes it. "Switch when it hits max" only works for gradual growth, never for a big-payload spike — which is the actual OOM trigger.

The architecture the evidence points to

Prevent first; use resilience as the net — and recycle, never "wait."

LayerDoesEvidence it relies on
① Preventoff-heap binary + size/concurrency limits so the heap rarely nears the capspike is 0.5 s (D) — you must prevent, not react
② Multi-instance + LBa crash/recycle of one node isn't a total outagerestart = 6.5 s outage per node (C)
③ Recycle (not wait)drain → force GC / restart → return to poolidle doesn't reduce (A); force GC & restart do (B, C)
④ Auto-restartlast-resort recoveryneeded but ugly alone (C)
n8n-specific gotcha Multiple web/main instances need multi-main mode (N8N_MULTI_MAIN_SETUP_ENABLED, Enterprise) so active triggers/schedules don't fire on every instance (leader election). Workers scale freely without it.
