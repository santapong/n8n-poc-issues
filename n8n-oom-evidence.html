★ FIXHow to prevent — step by step

Ordered, copy-pasteable. Each step tags the exhibit (proof) it neutralizes below. ✅ proven on this lab

A · Protect the web node (ingestion)
1 Cap upload / payload size at the edge Exhibit A proven H
Oversized uploads are rejected before they touch the heap (nginx returns 413).
# nginx.conf
client_max_body_size 25m;
# n8n env
N8N_PAYLOAD_SIZE_MAX=25
N8N_FORMDATA_FILE_SIZE_MAX=25
2 Keep binary data off the heap Exhibit A/B
Store files externally (a reference stays in the execution, not the bytes).
N8N_DEFAULT_BINARY_DATA_MODE=database   # queue mode (or s3)
N8N_BINARY_DATA_TTL=60
3 Don't route big files through n8n at all Exhibit A/G
Best for large files: client uploads to GCS/S3 via a signed URL; n8n gets only the object reference (a few KB). Removes the heap crash and the DB bloat.
B · Protect the worker (execution)
4 Batch & don't materialize the dataset Exhibit C proven I
Heap tracks what you HOLD. Process in chunks and return only what's needed.
# in the workflow
Loop Over Items / Split In Batches   # e.g. 200 rows per run
Return a summary, not x items.
Set (keep only set) to drop fields early after each call.
5 Isolate Code nodes in external task runners Exhibit D
A runaway Code node kills the runner subprocess, not the worker — and fixes the runner memory leak.
N8N_RUNNERS_MODE=external
N8N_RUNNERS_AUTH_TOKEN=<random-secret>
# runner sidecar (n8nio/runners image), its own:
NODE_OPTIONS=--max-old-space-size=<limit>
6 Bound concurrency & size the heap Exhibit C
Heap needed ≈ concurrency × max_MB × 2.3. Keep the V8 cap below the container limit so you get a clean error, not a kernel kill (137).
n8n worker --concurrency=5
NODE_OPTIONS=--max-old-space-size=<below container mem_limit>
# many small workers > one big worker
C · Data lifecycle
7 Don't save & do prune executions Exhibit G
Kills the finish-serialize spike, the "open a big execution → main OOM" poison pill, and Postgres bloat — all at once.
EXECUTIONS_DATA_SAVE_ON_SUCCESS=none
EXECUTIONS_DATA_PRUNE=true
EXECUTIONS_DATA_MAX_AGE=336            # hours (14 days)
EXECUTIONS_DATA_PRUNE_MAX_COUNT=10000
D · Resilience (non-memory)
8 Redis noeviction + watch queue depth Exhibit J
Eviction silently drops Bull jobs. Watch bull:jobs:wait for backpressure; autoscale workers on it.
redis-server --maxmemory-policy noeviction
# monitor / autoscale on:
redis-cli LLEN bull:jobs:wait
N8N_METRICS_INCLUDE_QUEUE_METRICS=true
9 Timeouts + generous probe tolerances Exhibit K
Bounds async/IO runs (⚠️ can't preempt sync CPU — Exhibit K). Give liveness probes slack so a brief CPU spike doesn't restart the pod.
EXECUTIONS_TIMEOUT=300               # seconds
# k8s livenessProbe: timeoutSeconds 5-10, failureThreshold 3-5
10 Graceful shutdown + idempotent workflows Exhibit F
Crashed/restarted jobs re-run FROM SCRATCH (never resumed) — so make workflows safe to re-run.
N8N_GRACEFUL_SHUTDOWN_TIMEOUT=60
# k8s terminationGracePeriodSeconds >= that
# design idempotent: dedup keys / upserts
E · Observability
11 Metrics on ALL nodes + alerts
The worker was a blind spot until we enabled this. Alert before the heap hits the cap.
N8N_METRICS=true                     # on MAIN and EVERY worker
# scrape all; alert when nodejs_heap_size_used_bytes nears the cap

Exhibit DTask-runner isolation (auto-restart run)

Proves: a Code-node OOM kills the isolated JS Task Runner, not the worker; execution is marked error, container does not restart. CONFIRMED

Proves: when arrival ≫ throughput, jobs pile in Redis bull:jobs:wait and executions sit "Queued", then drain. CONFIRMED

All 12 webhooks returned 200 instantly; the executions queued and the single worker drained them one at a time.

Exhibit KTimeout cannot preempt synchronous CPU

Proves: EXECUTIONS_TIMEOUT=10 did NOT stop a 30 s CPU busy-loop at 10 s — it ran to completion, then was marked error. CONFIRMED

The timeout can't fire while the loop blocks the event loop — it only takes effect after the loop finishes.
# EXECUTIONS_TIMEOUT=10 on worker; fired a ~30s CPU busy-loop
EXECUTIONS_TIMEOUT=10   # confirmed set
   t+7s   status=running
   t+31s  status=running     # well past the 10s cap — not preempted
   t+57s  status=running
   t+60s  status=error

execution 25:  status=error   ran_seconds=56     # ran the full loop, THEN errored
