1Overview & Goal

The goal was to reproduce, on purpose, the failure where the n8n web (main) node crashes with "JavaScript heap out of memory" when it receives large file uploads — running in the same shape as production: queue mode, behind nginx, with Prometheus /metrics enabled, and the Node heap capped at 512 MB so the crash is fast and observable.

🎯 One-line result
Uploaded files are turned into base64 strings on the V8 heap before being enqueued. Enough concurrent uploads push the heap past 512 MB and Node aborts (exit 134) — confirmed, with a heap snapshot showing 77% of the heap was strings.

🔎 Crash signature — why this matters
exit 134 = SIGABRT: V8 aborted itself after printing the fatal error. OOMKilled=false means the kernel did not kill it — V8 hit its own 512 MB cap first. That's exactly the real-world symptom (a Node heap error, not a silent SIGKILL).

There are two memory regions, and --max-old-space-size only caps one of them.

💡 The file isn't what overflowed — its base64 text is
~77% of the heap was string. The raw upload Buffer sits off-heap (external) and isn't capped by --max-old-space-size. What hits the 512 MB cap is the base64 string + execution JSON that n8n builds to enqueue the job.

⚠️ Queue-mode double hit
In default mode the base64 execution data is pushed through Redis to the worker, which base64-decodes it into its own heap. So the same data pressures both the web node and the worker.

9Cloud Run Constraints

🚫 filesystem mode is doubly out on Cloud Run
1. Cloud Run's writable filesystem is in-memory (tmpfs) — writing to disk consumes the instance's RAM, so it just trades a clean heap error for a Cloud Run "memory limit exceeded" kill (137).
2. Instances are isolated & ephemeral — no shared/persistent disk for main + workers to share.
Cloud Run leverWhy it helps
--concurrency=1..4Caps simultaneous requests per instance → one instance can't be flooded with many big uploads.
instance memory + --max-old-space-sizeWithout the flag, Node auto-sizes the heap to ~75% of instance memory.
GCS (S3-compatible)The natural object store on GCP if you ever adopt external binary mode.

13Observability — can metrics show which workflow ate memory?

❌ No per-workflow memory metric exists
All memory gauges (n8n_nodejs_heap_size_used_bytes, n8n_process_resident_memory_bytes, …) are process-wide — none carry a {workflow=…} label, and no flag adds one. One process = one shared V8 heap with no per-workflow tagging.
⚠️ Gap found: the worker has NO metrics
N8N_METRICS isn't set on the worker and Prometheus only scrapes n8n-main — so today you're blind to worker memory, which is exactly where the [1..x] loop would OOM.

How to actually attribute a spike to a workflow

flowchart LR A["METRICS (Grafana)<br/>WHEN + WHICH node"]:::proc --> B["WORKER LOG at that time<br/>executionId / workflowId"]:::proc B --> C["executions / Postgres<br/>→ workflow name<br/>→ biggest execution_data rows"]:::db classDef proc fill:#dbeafe,stroke:#2563eb,color:#1e3a8a; classDef db fill:#dcfce7,stroke:#15803d,color:#14532d;
✅ The deterministic trick
Run workers at --concurrency=1 and scale out. Each worker runs one execution at a time, so that worker's whole-process heap equals that one execution's memory — a spike on worker-3 = exactly the workflow it's running.
(Requires: enable N8N_METRICS on the worker + add a worker scrape target.)

15Clearing the heap? & what auto-restart does (Q1 & Q2)

Q1 — Can you clear the heap from inside a workflow?  No

AttemptResult (measured)
global.gc()needs --expose-gc — not set (NODE_OPTIONS=--max-old-space-size=512)
process.memoryUsage() in a Code nodeReferenceError: process is not defined — Code runs in a locked-down vm sandbox
❌ You can't free it at runtime
No GC handle, and no process/global in the Code-node sandbox. n8n also retains every node's output for the run, so the execution's own data can't be released mid-run. The only lever is to not accumulate: batch (Loop Over Items / SplitInBatches), paginate, and avoid returning giant item sets.

💡 Two things to remember
1. The in-flight execution is always dropped — marked crashed/error. n8n does not checkpoint or resume a run mid-way.
2. Where the OOM lands decides whether the container even restarts: a Code-node OOM kills the isolated JS Task Runner subprocess (worker survives, restarts=0). But an OOM in the worker/main process itself (buffering an upload, or serializing a huge result to save) kills the container — and with restart: unless-stopped + a retried heavy job, that becomes a crash loop.

Per n8n's docs, N8N_DEFAULT_BINARY_DATA_MODE accepts default, filesystem, s3, and database — and for queue mode n8n recommends database (filesystem isn't supported with queue mode). Binary mode only affects binary files, not the JSON/array item data that causes most OOMs here.

19Resilience & scaling (beyond memory)

Researched across n8n docs, GitHub & the forum; two reproduced on this lab. Defaults reconciled across sources.

✅ Reproduced — Queue backpressure

12 jobs (~5 s each) fired at a --concurrency=1 worker. All webhooks returned 200 instantly (onReceived), but executions sat "Queued" while the single worker drained them one at a time:

baseline bull:jobs:wait = 0
FIRE 12  → queued(new)=12,  bull:jobs:wait peaks at 12,  running=1
drain    → 12 → 11 → 10 → 9 → 8 → 7 → 6 → 4 → 3 → 2 → 1 → 0   (~1 job / 5s)

Lesson: arrival ≫ throughput → the Redis bull:jobs:wait list length is your backpressure gauge. Fix = more workers / higher concurrency (autoscale on that list).

✅ Reproduced — Execution timeout can't preempt sync CPU

SetupExpectedActual (measured)
EXECUTIONS_TIMEOUT=10, run a 30 s CPU busy-loopkilled at ~10 sran ~56 s, then marked error — not preempted
⚠️ Timeouts bound bookkeeping, not synchronous CPU
n8n is single-threaded — a busy for/while loop blocks the event loop, so the timeout callback can't fire until the loop finishes. EXECUTIONS_TIMEOUT works for async/IO-bound waits, but a CPU-bound loop runs to completion first. Same root as event-loop blocking below.
