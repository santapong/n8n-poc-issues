Headline: On this version the OOM you were worried about did not land where the lab's older notes predicted. The large-file upload no longer crashes the web node — n8n 2.28.6 streams binary to storage instead of piling base64 onto the main node's heap. The heap crash instead reproduces on the worker, and specifically inside the isolated JS Task Runner subprocess that executes Code nodes, when a workflow materialises a large array/JSON in memory.

2 · Where the data lives in Postgres

n8n created 110 tables. The ones on the execution path:

TableHoldsNotable columns (v2.28.6)
workflow_entityThe workflow the editor savesnodes, connections, active, pinData (all JSON)
webhook_entityRegistered webhook routeswebhookPath, method, workflowId
execution_entityOne row per runstatus, mode, storedAt (db/fs/s3/az), jsonSizeBytes, binaryDataSizeBytes
execution_dataThe heavy run payloaddata (text), workflowData (json)
binary_dataUploaded files as blobsdata bytea, fileName, mimeType, fileSize, sourceType
Key schema change vs the old lab notes: v2.28.6 has a dedicated binary_data table (raw bytea) and per-execution storedAt. Binary is stored by reference, not base64-inlined into the execution JSON on the heap — which is exactly why the upload no longer OOMs the main node (see §3).

Connecting to the database

Database name is n8n. Credentials come from .env (POSTGRES_USER / POSTGRES_PASSWORD / POSTGRES_DB). The postgres service is not published to the host, so use one of:

# A — interactive psql inside the container (quickest)
docker compose exec postgres psql -U n8n -d n8n
#   then: \dt  (list tables) · \d execution_entity  (describe) · \q  (quit)

# B — one-off query, no shell
docker compose exec -T postgres psql -U n8n -d n8n -c "SELECT * FROM execution_entity;"

For a GUI tool (DBeaver / TablePlus / pgAdmin) from the host, first publish the port — add ports: ["5432:5432"] to the postgres service and docker compose up -d postgres — then connect:

3 · Upload path — the web node survived

Reverted nginx to the vulnerable config (client_max_body_size 0, proxy_request_buffering off), imported and activated upload-oom so POST /webhook/upload was live, then flooded it:

SIZE_MB=250 CONCURRENCY=10 ROUNDS=6 ./scripts/upload-oom.sh

4 · Generate path — the worker crashed

The heap OOM reproduces where large data is materialised in memory. The generate-items workflow's Code node builds an array of x objects on the worker:

// Build x items on the WORKER (production execution)
const out = [];
for (let i = 0; i < x; i++) out.push({ json: { n: i, tag: 'item-' + i } });
return out;
# one request, five million items
curl -X POST "http://localhost:8080/webhook/generate?x=5000000"   → HTTP 200 (enqueued)

The crash signature (worker logs)

n8n-worker-1 | [30:0x7fc7…] 89636 ms: Mark-Compact (reduce) 559.5 (564.2) -> 525.2 (528.6) MB …
n8n-worker-1 | FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory
n8n-worker-1 | Node ran out of memory
n8n-worker-1 | Worker finished execution 2 (job 2)
n8n-worker-1 | Registered runner "JS Task Runner" (…)   ← runner subprocess restarts
The important architectural nuance: Code nodes in v2.28.6 execute inside an isolated JS Task Runner subprocess with its own heap — not in the worker's main process. So the OOM:
  • killed the runner, not the worker container → RestartCount stayed 0, OOMKilled=false;
  • was invisible in /metrics (which reports the worker's main process, which sat flat at ~125 MB) — only docker stats on the whole container and the crash log reveal it;
  • marked the execution status = error ("Node ran out of memory") and n8n kept serving. Reproduced twice — executions 2 and 3, each ran ~5.4 s before dying.

6 · What to do about it

RiskMitigation
Large file uploadsAlready handled on 2.28.6 (binary → binary_data). Keep an edge cap (client_max_body_size 25m) + N8N_PAYLOAD_SIZE_MAX as defence-in-depth. For big files, prefer upload-to-GCS + pass a reference so bytes never enter n8n or the DB.
Code nodes building huge arrays/JSONThe real remaining OOM. Process in batches (Loop Over Items / SplitInBatches), stream, or aggregate to a running total instead of returning N items — see generate-summary, which handles the same x with flat heap.
Runner crash blast radiusRunner OOM fails one execution, not the whole worker — good. Size N8N_RUNNERS_MAX_OLD_SPACE_SIZE deliberately and run runners in external mode for isolation in production.
Blind spotThe runner heap is not in /metrics. Alert on container RSS and on executions with status=error / "ran out of memory", not just nodejs_heap_size_used_bytes.
Governing formula (worker/runner): heap ≈ items × bytes_per_item. At ~100 B/item, 5 M items ≈ 500 MB → over the 512 cap. Bound the item count or the per-item size, or raise the runner heap to fit.
