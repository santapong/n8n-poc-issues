1 · Executive summary

The n8n web node and worker can be driven to a JavaScript heap out-of-memory crash under large or concurrent payloads. We reproduced both, measured them live, and traced the mechanism to the object level. The core cause is that payload bytes transit the Node/V8 heap (uploaded files become a base64 string on the heap; Code-node arrays materialize on the runner heap), and demand scales as concurrency × size.

Top findings

  1. Two distinct failure modes. A caught OOM (in a child JS Task Runner or a downstream service) is recorded as status=error; a hard OOM of the n8n process itself loses the execution entirely. That is the whole "sometimes it stores, sometimes it doesn't."
  2. "OutOfMemoryError: Java heap space" is not n8n. Node reports JavaScript heap out of memory; a JVM message means a downstream Java service your workflow calls ran out of heap — a separate problem.
  3. Binary uploads live on the heap as base64 in default mode (~1.37× file size), then are written into execution_data. This is the upload OOM path.
  4. Prevention needs both terms of the formula. Size limits alone are insufficient; concurrency must also be bounded, and — best of all — large binary kept off the heap entirely.
Headline recommendation Bound size (nginx + n8n) and concurrency, move large binary off-heap (filesystem/S3 or object-storage upload), size the heap from the formula heap ≥ concurrency × max_MB × 2.3, and fix the downstream Java service independently. "Scale up" is a derived number, not a standalone fix.

How an uploaded file is stored (proven)

In default binary mode the file is base64-encoded inline into execution_data.data — counted in jsonSizeBytes, with binaryDataSizeBytes=0 and the binary_data table empty (that table is used only by fs/s3/db-reference modes). Verified with a real 90 MB PDF:

Conclusion A successful upload stores the file completely and losslessly (base64 in execution_data). A crashed upload stores nothing — see §8.

5 · OOM reproductions (measured)

5a · Web node — concurrent large JSON → kernel cgroup kill

Eight concurrent 82 MB JSON bodies against /webhook/upload. Container RSS (raw bodies live off-heap; parsed objects on-heap) slams into the 1024 MB container limit and the kernel OOM-killer fires.

5b · Worker / JS Task Runner — 5 M-item array → V8 abort

POST /webhook/generate?x=5000000. A Code node builds a 5-million-object array on the runner heap.

Docker quirk OOMKilled can read false even when the kernel cgroup OOM-killer fired, because Docker only reflects a kill of container PID 1 (tini), not of the node child. dmesg is authoritative.

8 · RCA — why executions sometimes store and sometimes don't

The execution row is written by the n8n main/worker process. The single deciding factor is: did that process survive the OOM?

"OutOfMemoryError: Java heap space" is not n8n Node reports JavaScript heap out of memory / Node ran out of memory. A JVM message means a downstream Java service your workflow calls (an HTTP Request node → a Java API/microservice, a Java data node) ran out of heap and returned the text. n8n caught it and stored the run as error — which is precisely why those always store, while the runs that vanish are a different OOM: n8n's own process dying. You likely have two separate OOMs.

9 · Prevention & remediation

Governing formula peak heap ≈ concurrency × max_payload_MB × ~2.3  (uploads) — and separately runner heap ≈ items × bytes_per_item (Code nodes). Bound both factors, or the ceiling always loses.

Verdict The two ideas are the right start but not sufficient alone. Size limits + concurrency limits together are the core availability fix; scale-up is the number you derive from the formula, not a substitute for it; and the Java errors are a separate fix on the Java side.

10 · Observability tooling built

  • heap-monitor (:8888) — live per-node V8 heap + container RSS + per-V8-space composition; on-demand raw-heap capture (snapshot → parsed types & biggest strings) with a PDF-only filter and a timed (delayed) capture to catch transient payloads.
  • Heap snapshots — --heapsnapshot-signal=SIGUSR2 + --heapsnapshot-near-heap-limit=3; scripts/heap-snapshot.sh captures & copies out for Chrome DevTools.
  • scripts/heap-top.py — CLI snapshot parser (types, constructors, raw strings; --pdf filter).
  • Test assets — workflows/upload-test.json (echoes file metadata); real PDFs small/medium/max/oom.pdf via scripts/make-pdf.py.

11 · Recommendations & action plan

  1. Now (config, no infra change): nginx client_max_body_size 25m + limit_req/limit_conn; N8N_PAYLOAD_SIZE_MAX / N8N_FORMDATA_FILE_SIZE_MAX to your real max; cap worker concurrency. Size the heap from the formula.
  2. Next (removes the class of bug): move binary off-heap — filesystem/s3 mode, or an object-storage upload pattern where n8n only receives a reference.
  3. Workflow hygiene: batch/stream Code nodes; never return millions of items; paginate DB/HTTP reads.
  4. Separately: fix the downstream Java service heap (its -Xmx / pagination) — n8n limits won't touch it.
  5. Resilience: keep restart + health checks; alert on container RSS and on executions with status=error / "ran out of memory" (the runner OOM is invisible to /metrics).
